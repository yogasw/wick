1 · Sekarang vs Target

Sekarang — Monolith in-process

Call ~ns. Tapi: tambah connector = recompile core, crash 1 = mati semua, RAM konstan walau idle.

Target — Plugin eksternal via gRPC

Call ~0.1ms. Release independen, crash terisolasi, polyglot, marketplace, RAM idle 0.

Narik dari GitHub Releases API

OpsiCara
A. Registry repo (rekomendasi)1 repo indeks (wick-connectors); GET /releases → daftar + asset. Terkontrol, bisa di-sign.
B. Topic discoverySearch topic:wick-connector → tiap repo /releases/latest. Terbuka, tapi trust/rate-limit berat.
Marketplace = metadata, ngk download binary. list tarik nama+versi. Liat operasi available = tarik asset plugin.json (~KB). Binary di-download cuma saat Install.
Cache + rate-limit: GitHub API limit ketat → cache Releases (~15 mnt) pakai ETag + refresh manual/berkala. Reuse pola CatalogRefresh() yg udah ada (lihat §13).

Manifest INSTALLED didaftar saat startup-scan + hot-reload — bukan saat spawn. AVAILABLE ngk masuk registry sampai diinstall. Proses baru hidup saat execute.

7 · Spawn policy

LAZY (default)

execute pertama → spawn (~10-50ms sekali) → idle timeout → kill → execute lagi → spawn ulang. RAM idle = 0.

EAGER

spawn saat startup → selalu warm. Buat connector super-hot.

WARM POOL

pool N proses, scale naik-turun ikut traffic.
Proteksi resource (Termux RAM terbatas): cap concurrent plugin (mis. max 5) · idle timeout pendek · LRU evict saat RAM mepet.

8 · Untung vs Rugi (jujur)

UNTUNG
Build & release terpisah, versioning independen
Marketplace / download on-demand
Isolasi crash (panic ≠ mati core)
Polyglot (Go/Rust/Python)
Sandbox keamanan per-proses
Kontribusi pihak ketiga tanpa merge core
Core ramping, startup cepat, RAM idle rendah
RUGI
Overhead IPC ~10-200µs/call (ketutup API)
Versi proto host↔plugin harus cocok
Lifecycle kompleks (spawn/health/evict)
Build matrix OS×arch (wajib linux/arm64)
Supply-chain → wajib signing + checksum
RAM saat banyak plugin warm (tiap proc ~10-30MB)
Disk lebih besar (tiap binary bawa runtime)

9 · Jadi enteng & cepet? (analisis jujur)

ENTENG ✅
core binary kecil, startup cepat, RAM idle rendah, list/get zero-proses
SAMA ≈
throughput per-call (I/O-bound, IPC 0.1ms ketutup API 50-500ms)
BERAT ❌
RAM saat banyak plugin warm barengan, disk total

Realita pemakaian

punya 20 connector terinstall
   → dipakai per sesi: ~2-3 (github, slack)
   → punya proses hidup: cuma yang baru/lagi execute
   → sisanya: 0 proses, 0 RAM (cuma manifest di disk)
Connector dipanggil sporadis. Dengan idle-kill, model plugin sering lebih hemat RAM dari monolith — yang nganggur beneran nol, bukan nempel di core. Ngk semua plugin jalan bareng.
Worst case = burst (workflow fan-out ke banyak API). Dijaga: cap concurrent + idle timeout + LRU evict.

Kesimpulan: bukan "tiap call lebih ngebut" — tapi lebih modular + core ramping + skalabel + RAM idle hemat. Untung utama: release independen, marketplace, isolasi crash, komunitas.

11 · Latency — kenapa overhead ngk kerasa

Connector I/O-bound (nunggu API). IPC porsi kecil — asal proses persisten.

func call in-proc
~0s
gRPC UDS lokal
~0.1ms
gRPC TCP loopback
~0.3ms
spawn per-call ❌
~10-50ms
API GitHub (dominan)
~50-500ms
gRPC 0.1ms vs API 50-500ms = <0.5%. Ngk terasa — asal proses persisten, bukan spawn per-call.

10 trik tutup overhead

1Persistent subprocess

spawn sekali, pakai ribuan call.

2Unix Domain Socket

skip stack TCP/IP, ~2-5× lebih cepat dari TCP localhost.

3Connection reuse

gRPC HTTP/2 multiplex 1 koneksi.

4Schema caching

Schema() sekali saat load, cache host.

5Streaming RPC

list besar pakai server-streaming.

6Hybrid fast-path

connector hot tetap in-process.

7Warm pool

plugin sering dipakai dijaga warm.

8Shared memory

payload besar lewat mmap, kirim handle.

9Batching

op kecil digabung ExecuteBatch.

10Protobuf binary

encode cepat, jangan double-encode JSON.

12 · Kontrak proto & keamanan

Snippet lengkap & compile-able (semua message didefinisi di PLAN.md §12).

service Connector {
  rpc Schema   (SchemaRequest)   returns (SchemaResponse);
  rpc Execute  (ExecuteRequest)  returns (ExecuteResponse);
  rpc ExecuteStream (ExecuteRequest) returns (stream Chunk);   // bulk
  rpc Validate (ValidateRequest) returns (ValidateResponse);
  rpc Health   (HealthRequest)   returns (HealthResponse);
}
message Error { string code = 1; string message = 2; }
message ExecuteRequest {
  string operation = 1;            // "issues.create"
  bytes  args_json = 2;
  map<string,string> creds = 3;     // host decrypt wick_enc_ -> plain
  string request_id = 4;
  string session_id = 5;           // jaga scoping session wick
}
message ExecuteResponse { bytes result_json = 1; Error error = 2; }
message Chunk { bytes data = 1; bool eof = 2; Error error = 3; }
message SchemaResponse  { bytes manifest_json = 1; }   // dipakai --dump-manifest
message HealthResponse  { bool healthy = 1; string detail = 2; }
// SchemaRequest / ValidateRequest+Response / HealthRequest — lihat PLAN.md

Credential

Host decrypt wick_enc_, kirim plain lewat UDS+mTLS. Plugin ngk pegang key.

Verifikasi

sha256 + signature (cosign/minisign) sebelum exec.

Sandbox

user/cgroup terbatas, FS read-only, network allow-list.

Versi proto

handshake tolak plugin incompatible.

Dashboard config

tetap di host. Plugin cuma deklarasi field via Schema().

Registry

GitHub Releases + index JSON dulu. Registry sendiri nanti.

🔑 Temuan kunci: wick UDAH punya connector eksternal

Wick sudah punya jalur connector di luar core — custom MCP connector: core ngobrol JSON-RPC over HTTP ke MCP server remote.

internal/connectors/custom/mcp_client.goclient JSON-RPC, buildHeaders(), decrypt() buka wick_enc_
internal/connectors/custom/service.godefinisi custom: manual (HTTP) & mcp (server eksternal)

Plugin-gRPC = saudara LOKAL dari yang sudah ada. Custom-MCP = connector eksternal remote (HTTP). Plugin-gRPC = connector eksternal lokal (subprocess + UDS). Seam sudah ada — tinggal tambah transport, bukan bongkar arsitektur.

Kontrak yang dipertahankan — pkg/connector/connector.go

type ExecuteFunc func(c *Ctx) (any, error)
type Operation struct { Key, Name, Description string; Input []entity.Config;
                        Execute ExecuteFunc; Destructive bool }
type Category struct { Title, Description string; Ops []Operation }
type Module struct { Meta Meta; Configs []entity.Config; Operations []Category;
                     HealthCheck HealthCheckFunc; OAuth *OAuthMeta }

Module memetakan ke plugin.json apa adanya — perhatikan Operations itu []Category (operasi nested dalam category), operasi datar via mod.AllOps(), BUKAN array Operation datar. Flag LiveCatalog sudah ada untuk catalog dinamis — plugin manifest pakai mekanisme sama (sumber = plugin.json statis, bukan probe live).

🔌 Plug-and-play: reference connector.Module, bukan struct manifest paralel

manifest.json ──unmarshal──► connector.Module ──Register()──► [registry, MCP, seed, masking jalan semua]
                                  ▲ inject Execute = gRPC stub per-op (1 field runtime)

mod := connector.Module{}
json.Unmarshal(manifest, &mod)          // Meta + Operations + Configs(default) terisi
for op := range mod.AllOps() { op.Execute = grpcStub(conn, op.Key) }
registry.Register(mod)                    // SAMA kayak builtin → hilir ngk berubah
Register = Module UTUH, bukan cuma Meta. Default configs ikut di Module → Bootstrap → seedModuleRows() seed row config ke DB (persis alur builtin sekarang). Plugin nebeng seed yg sama, ngk bikin baru.
Plug-and-play sejati = shared types module: publish subset data-type pkg/connector (Meta/Operation/Configs) + proto stub jadi 1 Go module berversi. Core & plugin import yg sama → --dump-manifest = json.Marshal(mod) byte-identik → mustahil drift. Cuma Execute (Go func) yg ngk ke-serialize, di-inject runtime.

15 · Platform generik — connector → tools → jobs

Scope sekarang = connector. Tapi infra dirancang generik biar tools & jobs nyusul tanpa bikin ulang.

Shared (90% kode)

Semua di kotak hijau. Bikin sekali, dipakai semua kind.

Per-kind (10%)

17 · CI/CD auto-build & release

Connector berubah → auto rebuild, artifact otomatis dinamai nama-versi-os-arch, manifest digenerate, attach ke GitHub Release.

Repo connector (monorepo)

wick-connectors/        # tiap subfolder = 1 connector
├── github/  { go.mod, *.go, VERSION }
├── slack/   { go.mod, *.go, VERSION }
└── .github/workflows/release.yml
CI path-filter: cuma connector yg foldernya berubah yg di-rebuild.

Pipeline (GitHub Actions)

on: push paths: ['*/**']
detect:  git diff → folder berubah
build (matrix: name × os_arch):
  go build -o ${name}-${VER}-${OS}-${ARCH}
  ./${name}-... --dump-manifest > plugin.json
  sha256sum + cosign sign
release:
  gh release create "${name}/v${VER}" \
    ${name}-${VER}-*-* plugin.json *.sha256 *.sig

Konvensi nama artifact (otomatis)

github-1.4.2-linux-arm64      ← <name>-<version>-<goos>-<goarch>
github-1.4.2-linux-amd64
github-1.4.2-darwin-arm64
plugin.json                   ← manifest (generate dari binary)
github-1.4.2-linux-arm64.sha256 / .sig
Host pas Install pilih asset cocok dgn device (Termux → linux-arm64). Nama+versi terbaca dari nama file & plugin.json, ngk ditulis tangan.
Nyambung ke marketplace (§6): gh release create → muncul di GitHub Releases API → otomatis jadi entri Available di marketplace. Push connector → CI build+release → langsung nongol. Zero langkah manual. Matrix os/arch jalan paralel + path-filter → cepat & hemat CI.
