proposed internal/docs/workflow-design.md

Workflow

Multi-step automation graph yang bisa dibuat admin atau AI tanpa recompile. Drag-drop canvas + node-based: klasifikasi AI, skill, shell, HTTP, DB query, branch. Trigger cron / channel / webhook / manual. State file-based + resume-able.

canvas editor DAG-capable engine AI-editable via MCP test mode + fixtures resume on crash AI guard

1. Anatomi sebuah workflow

Empat sisi: trigger (kapan), action (apa), env (config + secret), datasets (state lookup). Notifikasi success/failure = explicit action node di graph (tidak ada field notify: terpisah). Plus governance (approve, guard). Engine impl pakai istilah "graph" / "node" / "DAG" — accurate di technical level (lihat §6).

1 — Trigger

Kapan jalan

  • cron
  • channel
  • webhook
  • manual
  • schedule_at
  • error n8n-style
2 — Action

Apa yg dilakukan (node types)

  • classify (AI)
  • agent (skills inside)
  • channel — outbound (reply/send/react)
  • connector — external API
  • shell / python
  • http / db_query
  • dataset_* — wick internal
  • branch / parallel
3 — Env & Secret

Config (§8)

  • text / textarea
  • secret (encrypted)
  • number / checkbox
  • dropdown / kvlist / picker
  • reuse config-tags vocab
4 — Datasets

Data store binding (§9)

  • datasets: field declare alias
  • shared wick_datasets_rows table
  • schema di dataset.yaml
  • UI tab terpisah (table view)
  • akses via dataset_* node di Action
Notifikasi via action node, bukan field terpisah. Workflow yang mau post Slack on-success: tambah type: channel atau type: connector action node di akhir graph. Branching success/failure pakai on_failure: fallback + fallback node. Implicit reply-to-source thread untuk trigger channel masih ada (engine convenience).
Error handling cross-workflow (n8n-style). Workflow declare on_error: { trigger_workflow: "error-handler" } → kalau fail, engine fire workflow "error-handler" yg subscribe type: error trigger. Error-handler dapet payload (source workflow, failed node, error msg, state snapshot). Loop protection: max 3 nested error workflows. Cocok buat centralized escalation + auto-retry pattern.

2. Use cases yang dicover

Empat pattern konkret. Workflow yang ga muat ke salah satunya = scope creep.

tree

1. Slack support triage

Pesan masuk #support → klasifikasi bug/question/feature → route ke connector / agent.

trigger: channel #support
  ↓
classify (AI): bug | question | feature | other
  ├─ bug      → connector: linear · create_issue   → channel reply
  ├─ question → agent: lookup docs + reply         → channel reply
  ├─ feature  → connector: airtable · append_row   → channel reply
  └─ other    → agent: friendly bounce
DAG

2. Incident response (paralel + merge)

Pager fires → fetch 3 sumber paralel → gabung → analisis AI → escalate/heal.

trigger: webhook /hooks/pagerduty
  ↓
parallel:
  ├─ http: grafana dashboard
  ├─ shell: ssh prod "df -h"
  └─ db_query: recent errors
  ↓ merge
agent: analisis
  ↓
classify: needs_human | self_heal
  ├─ human    → connector: pagerduty · escalate
  └─ self_heal → shell: runbook.sh
DAG

3. Daily digest

Cron pagi → fetch 3 sumber paralel → AI summarize → post Slack.

trigger: cron "0 8 * * *" Asia/Jakarta
  ↓
parallel:
  ├─ connector: github · list_issues (24h)
  ├─ connector: linear · list_issues (24h)
  └─ channel: slack · summarize #product-feedback
  ↓ merge
agent: bikin digest markdown
  ↓
channel: slack · send_message → #leadership
3 trigger → 1 workflow

4. Multi-stage interactive (ticket flow)

Inquiry → button → AI summary modal → submit ticket. 1 workflow, 3 trigger, 3 entry_node. Tidak ada engine pause — chain via channel interaction events.

triggers:
  - { type: channel, event: message,    entry_node: post-button }
  - { type: channel, event: action,     entry_node: fetch-thread }
  - { type: channel, event: submission, entry_node: create-ticket }

graph (3 stage chains):
  STAGE 1 (message → button):
    post-button (connector: chat.post_with_button)  [run #1 ends]

  STAGE 2 (action → AI + modal):
    fetch-thread → summarize (agent + skill) → show-modal  [run #2]

  STAGE 3 (submission → ticket + reply):
    create-ticket → confirm-reply  [run #3]
tree (nested)

5. Nested classification

Pesan → "ini pertanyaan?" → "topiknya apa?" → "open/closed?" → handler tepat.

trigger: channel chat #support
  ↓
classify-1: question | statement
  └─ question →
       classify-2: product-A | product-B | other
       └─ product-A →
            classify-3: how-to | bug-report
            ├─ how-to     → agent skills=[product-A-docs] → reply
            └─ bug-report → connector tracker.create + reply
Tree vs DAG. Tree (single parent, no merge) cukup buat 80% kasus AI orchestration. DAG (parallel + merge) perlu pas fan-out → gabung → process. Engine support keduanya; canvas bebas.

4. Katalog node types

Tiap node punya schema spesifik. Output di-stamp ke {{.Node.<id>.<field>}}, downstream consume via template.

CLS

AI

LLM klasifikasi → verdict enum. 6-layer reliability (lihat §5).

type: classify
provider: claude       # optional
preset: classifier-cheap
prompt: |
  Klasifikasi: {{.Event.Payload.text}}
output_cases: [bug, question]

# edges in graph.edges[]:
# - { from: , case: bug,     to: handle-bug }
# - { from: , case: default, to: end }
AGT

agent

provider-aware

Agent reasoning, skills inside (provider-specific bundle). Session: new / root / persistent.

type: agent
provider: claude          # claude|codex|gemini
preset: support-responder
session: new              # new|root|persistent
skills: [docs-search]
prompt: |
  Q: {{.Event.Payload.text}}
max_turns: 5
CHN

channel

agent domain

Outbound action via channel module (reply/send/react). Symmetric dgn trigger.

op: reply_thread

  thread: "{{.Event.Payload.thread}}"
  text: "Tracked: {{.Node.t.html_url}}"

connector

reuse existing

Invoke connector module op. Full reuse connectors/<name>/.

type: connector
module: github
op: create_issue
args:
  repo: "acme/inbox"
  title: "{{.Event.Payload.text}}"
# edge: { from: , to: reply-link }
SH

shell

Bash/sh exec. Output: .stdout, .exit_code, parse JSON optional.

  HOST: prod-1.abc.com

HTP

http

HTTP request dgn header/body template. SSRF allowlist enforced.

  Authorization: "Bearer {{.Secret.GH}}"

Parameterized SQL only. Output: .rows, .row_count.

  WHERE last_active > $1

  - "{{.Event.At}}"

Deterministic routing, no LLM. Boolean → if/else, string → switch.

type: branch

# - { from: , case: critical, to: page-oncall }

Fan-out N branches, wait-for-all. Output keyed per branch.

  - grafana-fetch
  - db-fetch
  - shell-check

# edge: { from: , to: enrich }

  DATA: "{{.Node.fetch.json}}"

Explicit fan-in. Wait listed nodes, combine output.

5. Reliable AI classification

LLM ga deterministik — output bisa "Bug.", "I think this is bug", atau "bug_report". Wick handle lewat 6 lapis defense. Default sudah agresif (layer 1-3 active), sisanya opt-in.

# Layer Apa yg dilakukan Default
1 structured_output AI di-prompt return JSON schema {verdict, confidence, reasoning}. CLI output di-parse + validate. Provider abstract per impl (lihat §5.1 CLI vs API SDK). ON
2 normalize Lowercase, trim, strip ./,/"/'. "Bug." → "bug". ON
3 exact match Cek cases keys. Found? Route. Skip ke layer 4. ON
4 fuzzy_match Levenshtein < 3 ke key terdekat. "bug_report" → "bug", "kestion" → "question". opt-in
5 retry_on_mismatch Output ga match? Re-prompt dgn pesan lebih strict ("JAWAB CUMA satu dari: ..."). Retry N kali. 1×
6 confidence_threshold AI return {verdict, confidence}. Kalau confidence < threshold → fire default case + audit warn. 0.0

Schema-enforced output

Engine derive JSON schema dari cases keys. Prompt builder kirim schema ke CLI sebagai instruksi, parser validate stdout:

{
  "name": "classify_response",
  "input_schema": {
    "type": "object",
    "properties": {
      "verdict": {
        "type": "string",
        "enum": ["bug", "question",
                 "feature_request", "other"]
      },
      "confidence": {"type": "number"},
      "reasoning": {"type": "string"}
    },
    "required": ["verdict"]
  }
}

Few-shot boost (optional)

Embed sample input → output di prompt, reliability naik signifikan:

examples:
  - input: "ada bug di chat widget"
    output: bug
  - input: "gimana cara connect?"
    output: question
  - input: "tolong tambahin dark mode"
    output: feature_request
End-state guarantee: Habis 6 layer masih mismatch → fire default case + audit log warning. default ga ada di cases = config bug, validator reject di parse time.

5.1 Structured output via CLI (bukan API SDK)

Wick agent jalan via subprocess internal/agents/provider/ — claude, codex, gemini adalah CLI binary, bukan API SDK call. Beda penting buat reliability strategy.

API SDK (n8n / direct integration) CLI subprocess (wick)
Schema enforcement tool_use / response_format di API call — provider-level Prompt-based + parse stdout — application-level
Output channel JSON response body stdout (text atau --output-format json kalau CLI support)
Latency ~200-1000ms ~500-2000ms (proc spawn overhead)
Tool ecosystem per node config inherit dari agent session (file/bash/MCP) — built-in
Cost predictability tinggi (token count langsung) indirect (parse CLI usage report)
Streaming server-sent events stdin/stdout stream
Provider interface (extension)
type Provider interface {
  Name() string
  // ... existing agent methods

  // New for classify-style nodes:
  StructuredCall(ctx, prompt, schema, opts)
    (StructuredResult, error)
}

type StructuredResult struct {
  Raw    string          // stdout CLI
  Parsed map[string]any  // JSON-parsed
  OK     bool            // parse + validate
  Error  string
  Usage  Usage                // tokens, cost
}

Per-provider implementation

Claude Code CLI

claude --output-format json --print <prompt> + system prompt yang minta JSON shape. Parse stdout.

Codex CLI TBD

Cek CLI capabilities, kemungkinan prompt-only + parse fallback.

Gemini CLI TBD

Cek CLI capabilities, kemungkinan prompt-only + parse fallback.

CLI ga support strict JSON mode → fallback prompt-only + agresif layer 4-5 (fuzzy + retry).

Universal baseline. Prompt-based JSON shape + parser (layer 1) bekerja di semua CLI. Per-provider Capabilities() interface declare structured mode support — engine pilih path optimal. Layer 4-5 tetep aktif sebagai safety net.
Trade-off accepted: wick ~1-2s per classify call (vs ~500ms API SDK), tapi agent dapet tool ecosystem (Read/Edit/Bash/MCP) built-in untuk node agent complex. Cocok untuk wick's domain: AI orchestration, bukan high-throughput API gateway.

5.3 Cost cap — defense terhadap channel spam

Classify call = subprocess CLI + LLM tokens. Channel-rame workflow bisa nuclear cost. Three-layer defense:

1 — Rate limit

60/min hard cap

Per workflow, existing §14 Security. Hit cap → trigger queue overflow per workflow config.

2 — Daily token budget

MAX_DAILY_CLASSIFY_TOKENS

Env field (widget: number, default 0=unlimited). Engine track running sum per UTC day. Hit cap → reject classify, fire default case if exists.

3 — Input cache (opt-in)

cache_ttl_sec

Per-node field. Hash Event.Text + prompt template, cache verdict. Saves N calls untuk pesan identik berulang. Reset on prompt change.

# workflow.yaml — apply cost cap
env:
  - name: MAX_DAILY_CLASSIFY_TOKENS
    widget: number
    default: 100000      # cap budget for spam protection

graph:
  nodes:
    - id: classify-intent
      type: classify
      cache_ttl_sec: 3600           # reuse classification 1h
      prompt: |
        ...
      cases:
        ...
        default: silent-end     # fires kalau budget exceeded

5.5 Session management — CLI subprocess reuse

CLI spawn ~500ms-1s overhead. Workflow dgn 5 agent node = 5× spawn kalau setiap node fresh. Field session: per node control behavior.

Mode Behavior Use case
new default Fresh subprocess per node, ga share context Isolation, deterministic, paralel-safe
root Share single subprocess di-spawn di awal workflow run, semua root-session node lewat proses sama. Sequential dalam run. Multi-turn reasoning across nodes, faster (1× spawn)
persistent Subprocess persist across workflow runs, session ID = workflow slug. Context inherit dari run sebelumnya Long-running assistant pattern, learn-from-history

YAML example

- id: classify-intent
  type: classify
  session: new           # default, classify independent

- id: deep-analysis
  type: agent
  session: root          # share context dgn root nodes lain
  ...

- id: ongoing-monitor
  type: agent
  session: persistent    # context bawa dari run sebelumnya
  ...

Engine behavior

  • state.json map: {sessions: {root: <proc_id>, persistent: <proc_id>}}
  • Root spawn lazy (saat node pertama dgn session: root run)
  • Workflow end → kill new + root sessions, detach persistent
  • Crash mid-run: state.json resume dari current, root re-spawn (lose intermediate context dari mode root — known trade-off, documented)
  • Parallel + root = serialize. Engine queue root-session nodes sequentially (1 proc, no concurrent prompts)
Default: classify = new (independent), agent = new (isolation safe). Opt-in root kalau butuh shared context, persistent selalu explicit.

7. Channel registry — generic interface

Channel (slack/telegram/rest/whatsapp/email) bukan hardcoded di workflow. Workflow konsumsi lewat registry self-describing — channel daftarin trigger spec + action spec, workflow auto-discover.

Channel interface
type Channel interface {
  Name() string
  IsConfigured() bool

  // Inbound: trigger schema
  TriggerSpecs() []TriggerSpec

  // Outbound: invoked via workflow node type: channel
  Actions() []ActionSpec
  Send(ctx, action, args) (any, error)

  // Event push to router
  Subscribe(handler EventHandler)
}

Action via type: channel (symmetric)

Channel Actions() diakses langsung via node type: channel. Ga auto-promote ke skill registry — channel = own domain (agent conversation), skill reserved untuk Claude Code skill bundle.

slack · reply_thread ← slack channel
slack · send_dm
slack · react
slack · create_canvas
telegram · send_message ← telegram
telegram · send_photo
rest · post ← rest channel
rest · put

Workflow consume: type: channel + channel: slack + op: reply_thread. AI discover via workflow_channels() MCP op.

For non-channel external API (GitHub, Stripe, Notion, ...) = pakai connector module. Beda domain (channel = agent conversation, connector = generic API). Workflow node type: connector + module: github + op: create_issue. Reuse 100% existing connector pattern.

Channel trigger with MatchSchema filter

triggers:
  - type: channel
    channel: slack
    event: message
    match_enabled: true   # gate; false = dump-all
    match:
      mode: whitelist
      channel_id: '[{"id":"C123","name":"general"}]'
      text_contains: "support"

Form rendered from EventDescriptor.MatchSchema — each event declares a wick-tagged Go struct via entity.StructToConfigs. Picker chips serialize as [{id,name}] JSON; router does id-membership at dispatch time.

Channel action node (symmetric trigger)

- id: reply-with-link
  type: channel            # same module as trigger
  channel: slack
  op: reply_thread     # one of channel.Actions()
  args:
    channel: "{{.Event.Payload.channel_id}}"
    thread:  "{{.Event.Payload.thread}}"
    text: "Tracked: {{.Node.ticket.url}}"

UI Inspector: pilih channel dropdown → op dropdown auto-populate dari Channel.Actions(). AI discover via workflow_channels().

Implicit reply-to-source. Kalau trigger dari channel dan workflow ga punya explicit type: channel action ke source thread, engine inject synthetic node di akhir flow dgn {{.Run.final_result}} ke thread asal. Override via reply_source: false di trigger spec.
Adding new channel. Bikin internal/agents/channels/<name>/, implement Channel interface (TriggerSpecs + Actions + Send), register di setup.Compose. Trigger router + action dispatcher otomatis pick up. Workflow engine + UI tidak butuh perubahan.

8. Environment & secrets — reuse config-tags vocab

Workflow butuh config: Slack channel target, GitHub PAT, retry, feature flag. Schema di workflow.yaml (developer contract, version-controlled). Values di <slug>/env.yaml (UI-managed, secrets encrypted dgn wick_enc_).

Reuse vocabulary wick:"..." config-tag yang sudah ada (docs/reference/config-tags.md) — same widget + modifier names, same form renderer, same secret/kvlist/picker UI behavior. Beda cuma: Go module schema di struct tag, workflow schema di YAML.

Schema di workflow.yaml

    required: true

  - name: MAX_DAILY_RUNS

    default: 100

  - name: ESCALATION_MODE
    widget: dropdown
    options: [pager, slack, email]
    default: slack

  - name: GUARD_PROMPT_EXTRA
    widget: textarea

  - name: ENABLE_AUTO_TRIAGE
    widget: checkbox        # auto for bool
    default: true

  - name: ALLOWED_CHANNELS      # multi-row table

Service translate ke parameterized SQL. Kontrak: no string concat, no raw SQL passthrough, no OR 1=1 injection bisa lewat.

Why no raw SQL?
  • Safety — workflow author bisa AI, ga trust raw SQL ke prod DB
  • Schema enforcement — query divalidasi terhadap dataset.yaml
  • Access control — per-dataset allowlist enforceable di Service layer
  • Index hint — Service tau index relevan, bisa optimize
  • Audit — tiap call log-able dgn structured args
  • Future-proof — kalau pindah ke Mongo/Redis, Service hide impl
Power user butuh raw SQL?
  • UI Query Console — audit-able, schema-aware, scoped ke 1 dataset
  • db_query node ke external read-replica DB
  • Direct DB access lewat connector — outside workflow scope

Sharing antar workflow — norm, bukan exception

Share dataset = use case utama (dedup events, cross-workflow state, shared lookup). Safety bukan dari no-share, tapi dari explicit contract.

4 safety layers buat shared dataset. Strictness, allowlist, row_filter, version pin. Default safe (strict + explicit). Power user opt-out kalau benar2 butuh JSONB flexibility.

Strictness modes

Mode Behavior Use case
strict default Reject insert kalau ada extra key di luar columns. Workflow harus declare semua fields Production, critical data, multi-workflow share
lax Accept extra keys, simpan ke JSONB. Warn di UI saat read kalau ada key tak dikenal. Query by extra key = full scan Dev iteration, exploration, ad-hoc data
extensible Per-workflow boleh tambah column. Schema = union. Engine track _meta.added_by_workflow Federation pattern, plug-in workflows

Migration safety — preview, dry-run, atomic, rollback

Schema change yg sentuh existing rows = formal flow dgn manual user trigger di tiap step. Never auto-magic.

Flow

  1. User edit dataset.yaml (UI form atau hand-edit)
  2. Engine compute diff vs current active version
  3. UI Preview panel — schema diff color-coded + data impact estimate ("1,247 rows touched")
  4. [Dry-run] — validate di shadow, report rows yg ga fit. No data touched
  5. [Apply] — typed confirmation kalau destructive ("type DROP to confirm")
  6. Migration job spawn (reuse JobRun runner), snapshot ke history/v<N+1>.yaml first
  7. Atomic per-batch (1000 rows default) dlm transaction, throttle 100ms
  8. Progress visible di JobRun + dataset detail. Pause/resume support
  9. Done → schema active, history snapshot final, audit log entry

Safety properties

Idempotent

Engine stamp _meta.migrated_to: v<N> per row. Re-run skip rows already migrated. Crash mid-job → restart aman.

Atomic per-batch

Partial fail rollback batch transaction. Mark migration failed di JobRun, user retry atau revert.

Rollback via snapshot

Sebelum apply, snapshot ke history/v<N+1>.yaml. Revert = swap dataset.yaml dgn old version + reverse migration job.

Throttle + observable

100ms between batches avoid lock contention. Pause/resume mid-flight. JobRun page show progress.

13. AI guard sebelum publish

Ephemeral reviewer agent baca semua node + script + prompt, banding dgn rule. Block / warn / off per config.

Destructive

No rm -rf, dd, mkfs, fork bombs.

Injection

classify prompt ga passthrough {{.Event.Payload.text}} ke shell raw. db_query parameterized only.

Network

HTTP host allowlist. No local IP (SSRF defense). Secret tag (wick_enc_) wajib.

14. Security model

Workflow boleh exec arbitrary code dan call external — surface area besar. Multi-layer defense.

Shell / Python — arbitrary exec
  • AI-generated → approved=false by default
  • UI Create wajib role admin
  • Approval modal show diff command + script
  • AI guard scan destructive pattern
Prompt injection via channel
  • Wrap user input dgn <user_input> tag
  • Whitelist user/group per trigger
  • Guard flag direct passthrough ke shell
DB / HTTP — injection & SSRF
  • db_query PAKSA parameterized ($1, $2)
  • HTTP host allowlist, block local IPs by default
  • Rate limit per workflow (60/min default)
Webhook / Manual
  • HMAC SHA-256 mandatory di webhook
  • IP CIDR allowlist optional
  • require_role: admin di manual trigger
  • Audit log catat clicker + replay actor

15. Differentiator vs n8n

Bukan kompetitor — pendekatan AI fundamentally beda. n8n = API SDK approach; wick = CLI subprocess approach.

Aspect n8n Wick Workflow
Decision logic If/Switch dgn expression branch (if/switch) + classify (AI natural lang) — dua-duanya tersedia
AI integration API SDK per node (OpenAI/Anthropic) — 1 node = 1 API call CLI subprocess (claude/codex/gemini) — AI = agent dgn tool ecosystem built-in (Read/Edit/Bash/MCP)
Output reliability tool_use schema enforcement di API level Prompt-based JSON + parser + 5-layer fallback (§5.1)
AI nodes Add-on, generic LLM call First-class, pool + session reuse, share state antar node
Editor UI wajib UI atau YAML atau AI-via-MCP
Storage DB File-based, gitops-friendly
Skill orchestration Generic webhook Skill registry first-class — channel actions auto-registered
Deployment Hosted / self-host k8s Embedded di wick binary
Latency per AI node ~200-1000ms (API direct) ~500-2000ms (proc spawn overhead) — accepted trade-off
n8n approach

Call OpenAI API per node, tiap call independen. Cocok buat "LLM as one of many service integrations".

Wick approach

Spawn agent CLI yang punya tool ecosystem (file/bash/MCP) inheriting dari agent session. Cocok buat "AI orchestration dgn skill + file manipulation + MCP tools".
