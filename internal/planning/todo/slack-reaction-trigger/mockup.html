Reaction Auto-Reply Switch

When a thread is started by @bot, the agent automatically arms auto-reply and drops a 🤖 on the parent (top) message. While the 🤖 stays, every new reply goes to the agent without a mention. The user removes 🤖 to stop and can re-add it to resume — no manual reaction needed to turn it on. The flag is saved in the session meta, so it survives a restart. Threads are still started by @bot only — auto-arm never opens a new session.

Flow — new @bot thread → AUTO-ARM (primary path)
1
@bot mention creates a new channel thread → handleMessage, the send creates the session on disk.
2
Gate. new session · channel (not DM) · ReactionTriggerEnabled · channel allowed (all = any, whitelist = in ReactionChannels).
3
Arm + mark. setAutoReply(true) persists Meta.AutoReply=true and the bot drops the 🤖 marker on the parent. No manual reaction needed.
Flow — new reply arrives while armed (auto-reply)
1
message event in the channel (existing handler, extended). Not a mention.
2
Resolve threadTS → sessionID. autoReplyOn? Cache hit → use it; miss → read Meta.AutoReply and warm the cache (this is what makes it survive restart). Not armed → today's DM-only behaviour.
3
Armed → re-check reply author passes allowedCfg, then handleMessage appends the reply as a user turn to the live session.
Flow — 🤖 removed (OFF) · re-added (manual re-arm)
1
reaction_removed on the parent (emoji + reactor ≠ bot) → setAutoReply(false): clears cache + persists Meta.AutoReply=false. Future replies go quiet.
2
No abort. A turn already in flight finishes and answers — only the next reply is dropped.
3
reaction_added by a user (passes allowedCfg, item is the parent, session exists) → setAutoReply(true) again. Re-arm is manual; the bot's own reaction is skipped.

Independent of the access whitelist. Only these channels react to 🤖.
🤖 robot_face fixed
Hardcoded for now. Configurable later if needed.
Required Slack scope
reactions:read
Without it, Slack never delivers reaction_added / reaction_removed — the feature is silently inert. Surface this in the Status panel.
Gate (add & remove)
  • ✅ emoji == robot_face
  • ✅ reactor ≠ bot (no self-trigger loop)
  • ✅ ReactionTriggerEnabled on
  • ✅ channel allowed — mode all (any) or whitelist (∈ ReactionChannels)
  • ✅ reactor passes allowedCfg (Users/Groups)
  • ✅ item is the parent message (not a reply bubble)
  • ✅ session already exists for the thread add only
A reaction is ambient, not a command — a failed gate is silently ignored (no "access denied" DM, unlike a real message). The switch governs future replies; it never sends the reacted message itself and never aborts a running turn.
Edge cases
🤖 on a reply bubble, not the parent — ignore. The switch only lives on the thread root.
Thread has no session (never @mentioned) — ignore. The switch never boots a thread.
Remove while a turn is running — that turn finishes & answers; only the next reply is dropped. No abort.
Disabled / channel not listed — silently ignore.
Bot restart — the autoReply set is in-memory; switches reset on restart. Re-react to re-arm. (Persist later if needed.)
Reply author not allowed — re-checked per reply via allowedCfg; a non-allowed member's reply is skipped even with the switch on.
